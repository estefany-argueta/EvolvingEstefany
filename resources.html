Reproducible Research Resources

This page collects starter templates, guidelines, and workflows that my lab uses to promote reproducible, well-documented science. Feel free to adapt these for your own work.

Reproducible Research Starter Kit

Folder & Repository Structure

Recommended project layout:

project-name/
├── README.md
├── data/
│   ├── raw/              # Never modify!
│   ├── processed/        # Cleaned, annotated data
│   └── metadata/
├── scripts/
│   ├── 01_data_import.R
│   ├── 02_clean_and_explore.R
│   └── 03_analysis.R
├── outputs/
│   ├── figures/
│   ├── tables/
│   └── models/
├── docs/
│   └── data_dictionary.md
└── .gitignore

Key principle: Raw data stays raw. Create new files for each processing step. Use numbered scripts to show the workflow.

README Template

Every project should have a README with:

  • Project title and brief description
  • Authors and affiliations
  • Folder structure overview
  • Data sources and collection methods
  • Instructions to reproduce analyses
  • Dependencies (R packages, software versions)
  • License information
  • Contact for questions

Data Organization & Naming

  • File naming: Use descriptive names with dates. Example: 2024-06-23_thermal_tolerance_trials.csv
  • Consistency: Use consistent column names, units, and missing value codes (NA, -9999, etc.)
  • Metadata: Always include a data dictionary explaining all columns, units, and collection protocols
  • Version control: If data changes, create a new file (v1, v2) rather than overwriting
  • Documentation: Record who collected the data, when, where, and under what conditions

Git & GitHub How-Tos

Branching & Workflow

  • Main branch: Production-ready code only. Protects published work.
  • Develop branch: Working branch where we test new analyses.
  • Feature branches: For experiments. Name them: feature/add-thermal-plots or bugfix/missing-samples
  • Pull requests: Always use PRs to merge back to develop/main. Require review before merging.
  • Commit messages: Be descriptive: Add thermal tolerance data for northern populations

Issues & Project Boards

  • Use Issues to track bugs, feature requests, and analysis questions
  • Label issues: bug, enhancement, analysis, priority
  • Use Project Boards to organize workflow (To Do → In Progress → Done)
  • Link PRs to issues: "Closes #42" in your PR description

R Workflows

Project Template

Start each R project with this structure at the top of your script:

# Analysis: [Brief description]
# Author: [Your name]
# Date: YYYY-MM-DD
# Notes: [Any important details]

# Load packages
library(tidyverse)
library(ggplot2)

# Source functions
source("scripts/shared_functions.R")

# Set seed for reproducibility
set.seed(12345)

# Load data
raw_data <- read.csv("data/raw/raw_data.csv")

Plotting Standards & Figure Checklist

Before finalizing any figure, check:

  • ☐ Axes are labeled with units
  • ☐ Legend is clear and positioned appropriately
  • ☐ Color scheme is colorblind-friendly
  • ☐ Font sizes are readable (especially in presentations)
  • ☐ Error bars or confidence intervals are documented
  • ☐ Figure caption is descriptive and complete
  • ☐ Resolution is sufficient for publication (300+ dpi)
  • ☐ Figure is saved in reproducible format (script + data)

Field & Lab Documentation

Field Datasheet Template Ideas

  • Include header with date, location, observer, weather conditions
  • Pre-populate species/sample IDs where possible to reduce errors
  • Use consistent units and measurement protocols
  • Include space for qualitative notes (behavior, conditions, concerns)
  • Always collect: date, time, location (GPS if possible), observer initials
  • Make datasheets waterproof or digitize immediately in the field

Metadata Checklist

Every dataset should include metadata documenting:

  • Data collection date(s) and location(s)
  • Who collected the data and who processed it
  • Collection protocol or methodology (methods paper reference)
  • All column names, units, and data types
  • Missing values and how they're coded
  • Any data quality issues or caveats
  • File version number and modification date
  • License and how the data can be used
